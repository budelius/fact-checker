Visual and interaction contract for TikTok ingestion and claim extraction. Generated by gsd-ui-researcher for validation by gsd-ui-checker.

| Property | Value |

| Preset | not applicable |

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, status dot offsets, compact timestamp spacing |
| sm | 8px | Compact element spacing, badges, button icon gaps |
| md | 16px | Default field spacing, form rows, artifact rows |
| lg | 24px | Panel padding, section separation, right rail groups |
| xl | 32px | Main pane padding, major workspace groups |
| 2xl | 48px | Empty state vertical spacing, large progress group separation |
| 3xl | 64px | Page-level spacing only when the shell is in a single-column mobile layout |

Exceptions:

- Existing `--topbar-height` remains 56px.
- Existing rail widths remain 248px left and 320px right unless Phase 2 content requires the right rail to expand to 360px at desktop widths.
- Existing desktop buttons may remain 36px high. New mobile controls must use a 44px minimum touch target.
- Screenshot and media previews must use stable dimensions with a 16:9 aspect ratio. Do not allow thumbnails to resize when status text changes.
- Border radius remains 8px maximum.

## Typography

Use exactly these four sizes for Phase 2 additions. Use only weights 400 and 600 in new UI.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 15px | 400 | 1.6 |
| Label | 12px | 600 | 1.35 |
| Heading | 24px | 600 | 1.25 |
| Display | 32px | 600 | 1.15 |

Rules:

- Use 13px mono text only for UUIDs, timestamps, route-like values, provider names, and raw artifact identifiers.
- Keep letter spacing at 0 except existing uppercase micro-labels, which may keep browser-default spacing with no negative tracking.
- Long URLs, UUIDs, transcript excerpts, and claim text must wrap with `overflow-wrap: anywhere`.
- Phase 2 headings should describe the current object or state, such as "TikTok ingestion", "Job status", "Extracted claims", or "Research basis".

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#f8faf7` | App background and broad workspace area |
| Secondary (30%) | `#ffffff` | Topbar, rails, form surface, artifact panels, transcript and claim rows |
| Accent (10%) | `#256f62` | Primary CTA, active progress step, active navigation item, focus rings, selected claim/artifact linkage |
| Destructive | `#b42318` | Failed job state and destructive actions only |

Accent reserved for:

- Primary CTA: `Extract claims`
- Active progress step indicator
- Active navigation/workspace item
- Focus-visible outlines
- Selected claim, transcript excerpt, or screenshot cross-reference
- Complete/success state accents when paired with text labels

Additional semantic colors from existing tokens:

- `#dceee9` accent-soft: selected row background, complete step background, linked artifact highlight.
- `#8a5a00` evidence/source clue: research-basis candidates, DOI/arXiv/URL chips, source-clue screenshot labels.
- `#60706a` muted: secondary metadata, pending status, explanatory empty-state body.
- `#d8e0dc` border: panel boundaries, artifact separators, step connectors.

Do not introduce a new palette in Phase 2. If a warning color is needed for "needs manual review", use `#8a5a00` with explicit text rather than adding orange variants.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Extract claims |
| Empty state heading | Paste a public TikTok URL |
| Empty state body | Start with a public video link to create a job, retrieve transcript context, capture source-clue frames, and extract timestamped claims. |
| Error state | This TikTok could not be processed. Check that the link is public, try another URL, or use the pasted-transcript fallback. |
| Destructive confirmation | none; Phase 2 must not add delete, purge, or destructive artifact actions |

Additional required labels:

| Element | Copy |
|---------|------|
| URL field label | Public TikTok URL |
| Job UUID label | Job UUID |
| Lifecycle label | Status |
| Current operation label | Current operation |
| Transcript unavailable | Transcript unavailable |
| Screenshot empty state | No source-clue frames captured yet. |
| Claims empty state | No claims extracted yet. |

## Phase 2 Interaction Contract

### Shell Placement

| Area | Contract |
|------|----------|
| Topbar | Keep brand on the left. Set section label to `TikTok ingestion` while the Phase 2 workspace is active. Keep `Browse vault` as a secondary command, not the main call to action. |
| Left rail | Preserve vault-oriented navigation. Add an ingestion workspace entry only if the implementation adds top-level workspace switching; otherwise keep the left rail unchanged. |
| Main pane | First viewport must show the URL submission form and current or empty job state. No marketing hero, no decorative preview card. |
| Right rail | Show job metadata, artifact counts, provenance, and selected claim/screenshot details. On mobile this content moves below the main pane. |

### Submission Flow

| State | Required UI |
|-------|-------------|
| Empty | URL input, primary `Extract claims` button, secondary development fallback entry point for pasted transcript or fixture ingestion. |
| Submitting | Disable duplicate submit for the same URL, keep the URL visible, show job creation in progress, and reserve space for the job UUID. |
| Job created | Show the job UUID in mono text with a copy-icon button and visible lifecycle status. |
| Running | Show the progress timeline, current operation, artifact statuses, and any retrieved metadata as soon as available. |
| Complete | Keep progress visible with all completed/skipped states, show transcript/media/claim artifacts, and show research-basis triage. |
| Failed | Keep the failed step visible, show recoverable error copy, preserve the entered URL, and provide `Try another URL` plus pasted-transcript fallback. |

The top-level job lifecycle must use the Phase 1 language: `pending`, `running`, `failed`, and `complete` or `succeeded`. If the backend returns `succeeded`, display `complete` in the UI while preserving raw status in job metadata.

### Progress Timeline

Use a vertical timeline on desktop and a compact stacked list on mobile. Do not use a generic spinner as the only progress indicator.

| Step | Display Label | User-Facing Operation Examples | Terminal Artifact |
|------|---------------|--------------------------------|-------------------|
| 1 | Validate URL | Checking public TikTok URL | valid URL or unsupported URL error |
| 2 | Read public metadata | Reading public page metadata and captions | video metadata, creator alias if available |
| 3 | Build transcript | Loading captions or transcribing allowed media | transcript artifact with provenance |
| 4 | Capture source clues | Capturing frames with paper titles, DOI/arXiv strings, URLs, slides, or article screenshots | screenshot/keyframe artifacts |
| 5 | Extract claims | Extracting atomic AI research claims from transcript context | claim artifacts |
| 6 | Triage research basis | Looking for paper-like titles, DOI/arXiv IDs, URLs, named studies, and source references | research-basis status and candidate list |
| 7 | Write owned artifacts | Writing normalized artifacts to the owned knowledge model | vault/MongoDB-ready artifact status |

Each step must show one of: `pending`, `running`, `complete`, `failed`, or `skipped`. `skipped` is allowed only for optional paths such as video download/transcription fallback or frame capture when unavailable.

### Artifact Status

Show a status grid or list for these artifacts in the main pane or right rail:

| Artifact | Required Status Values | Required Details |
|----------|------------------------|------------------|
| Public metadata | pending, running, complete, failed | source URL, external video ID if known, creator alias if known |
| Transcript | pending, running, complete, failed, unavailable | retrieval method, provider/source, segment count, quality note, failure reason |
| Media retrieval | pending, running, complete, skipped, failed | compliance gate result, media artifact UUID if created |
| Screenshots/keyframes | pending, running, complete, skipped, failed | count, timestamps, vault asset paths, source-clue flags |
| Claims | pending, running, complete, failed | count, extraction confidence summary, parse-failure state |
| Research basis | pending, running, complete, needs_manual_review | source candidate count and basis classification |

Render external transcript text, OCR text, public-page metadata, and visual text as plain text only. Never render untrusted external HTML.

### Transcript And Claim Review

| Element | Contract |
|---------|----------|
| Transcript preview | Segment list with timestamps, excerpt text, method/provenance badge, and active segment highlight. |
| Claim list | Each row shows claim UUID if available, timestamp or time range, atomic claim text, transcript excerpt link, screenshot refs, extraction confidence, and `evidence_status: pending`. |
| Claim selection | Selecting a claim highlights its transcript excerpt and related screenshot/keyframe previews. |
| Parse failure | Show the raw failure state and a retry-safe message. Do not fabricate claims from a failed parse. |
| Evidence labels | Do not show supported, contradicted, mixed, or insufficient evidence labels in Phase 2 claim rows. Those belong to Phase 4. |

### Screenshot And Source-Clue Preview

| Element | Contract |
|---------|----------|
| Thumbnail layout | 16:9 thumbnails in a horizontal strip or responsive grid, with fixed dimensions and no layout shift. |
| Labels | Show timestamp, screenshot UUID/path when available, and a `source clue` label when the frame contains paper/source signals. |
| Preview intent | Previews support memory and source discovery. They must not center the design around the creator's face or decorative video imagery. |
| Empty state | Use `No source-clue frames captured yet.` |
| Interaction | Selecting a screenshot shows related claims, OCR/source-clue text if available, and vault asset path in the right rail. |

### Research-Basis Triage

The research-basis panel is required and must appear even when no candidates are found.

| State | Display Copy | Required Details |
|-------|--------------|------------------|
| `source_candidates_found` | Source candidates found | Count and chips for DOI, arXiv, URL, paper title, named study, author, or article title candidates |
| `no_research_source_found` | No research source found | Explain that no paper-like references were found in transcript or screenshots |
| `opinion_or_unratable` | Opinion-based for now | Make clear that no evidence evaluation has been attempted |
| `needs_manual_review` | Needs manual review | Show ambiguous candidates and the reason the system could not classify them |

Candidate chips use the evidence/source clue color (`#8a5a00`) and must remain secondary to the top-level job status. The panel should say "ready for paper discovery" only when candidates exist; it must not imply that papers have already been found or evaluated.

### Recoverable Errors

| Error Type | Required Behavior |
|------------|-------------------|
| Unsupported URL | Preserve input, show unsupported URL copy, focus the URL field, and offer retry. |
| Private/unavailable video | Explain that only public TikTok links are supported in v1. |
| Captions unavailable | Continue to allowed transcription fallback or mark transcript unavailable with reason. |
| Compliance gate blocks media retrieval | Mark media retrieval as skipped or failed with a compliance-specific message; do not block transcript-based fixture or pasted-transcript development flow. |
| Claim extraction parse failure | Show failed claim extraction step, keep transcript artifact visible, and allow rerun without duplicate canonical entities. |
| Backend/job polling failure | Keep the job UUID visible and show a retry status action; do not clear job context. |

Use `role="alert"` for blocking errors and `aria-live="polite"` for progress updates.

## Accessibility And Responsive Contract

- URL submission must use a real `<form>` with a visible `Public TikTok URL` label.
- The primary CTA must be reachable by keyboard and submit on Enter when the URL field is focused.
- Job status changes must be announced with `aria-live="polite"`.
- Blocking errors must use `role="alert"`.
- Icon-only controls such as copy UUID, retry, expand, collapse, and preview must have accessible labels and tooltips.
- Focus-visible outline uses `#256f62` and 2px offset, matching the existing shell.
- Mobile layout below 720px keeps topbar stacked, vault navigation horizontal, main pane first, and right rail metadata below main content.
- Minimum mobile touch target is 44px for form controls, copy/retry buttons, and screenshot selection.
- Text must never overlap thumbnails, badges, or buttons. Use wrapping, fixed preview aspect ratios, and stable row heights for dynamic status text.
